Dashboard API Logs

Background Polling

This controls how often the server contacts NuHeat to fetch the latest thermostat data (current temperature, heating status, schedule mode). A shorter interval means fresher data on the dashboard, but sends more requests to NuHeat. The default of 300 seconds matches what Home Assistant and other integrations use.
Poll interval (seconds)
60 = poll every minute (fast updates, more API traffic)
300 = poll every 5 minutes (recommended default)
900 = poll every 15 minutes (minimal API traffic)

Rate Limits

Limits how many requests a single IP address can make to this server per minute. This protects against a misconfigured device or script flooding the server with requests. Read limits apply to status/list endpoints. Write limits apply to set-temperature, resume-schedule, and force-refresh endpoints. Requests that exceed the limit get an HTTP 429 response.
Read limit (per min per IP)
60 = one device can check status once per second (default)
10 = stricter, allows polling at most every 6 seconds
200 = more relaxed, for dashboards refreshing frequently
Write limit (per min per IP)
10 = allows 10 temperature changes per minute per device (default)
3 = stricter, good if you want to prevent accidental rapid changes
30 = more relaxed, for automation scripts that batch changes

Write Throttle

The minimum time the server waits between sending temperature change commands to NuHeat's API. This is a global throttle across all clients -- even if multiple devices or users try to change temperatures at the same time, commands are queued and spaced out by this delay. This prevents overwhelming NuHeat when several thermostats are adjusted in quick succession.
Throttle delay (seconds)
60 = one command to NuHeat per minute at most (safe default)
30 = faster response for multiple thermostat changes
0 = no throttle, send commands immediately (use with caution)
Changes take effect immediately without restarting the container. They revert to .env defaults on restart.
